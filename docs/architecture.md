# AirLink Controller 実装可能性・アーキテクチャ整理

## 前提と結論

このプロジェクトの目的は、**iPhone の画面を Wi-Fi 経由で Windows PC に表示すること**です。現状の `index.html` は UI モックであり、ブラウザ単体では iPhone 実機の画面ミラーリング受信・復号・表示はできません。

まずは **表示のみの MVP** として、iPhone の「画面ミラーリング」から Windows PC 側アプリを選び、Windows 上で映像を表示できる状態を目指します。PC のクリックで iPhone を完全操作する機能は、iOS の権限制限・プラットフォーム制約・セキュリティ観点により **MVP 対象外** とします。

将来的には Windows デスクトップアプリ化し、Web UI は Tauri または Electron のフロントエンドとして再利用します。

## 1. MVPスコープ

### MVPで実現すること

- Windows PC 上で受信アプリを起動できる。
- iPhone と Windows PC が同一 Wi-Fi / 同一 LAN にいる前提で、iPhone 側の画面ミラーリング候補に PC を表示する。
- iPhone から開始されたミラーリング映像を Windows PC で表示する。
- 受信状態、接続中端末名、解像度、フレームレート、エラーを UI に表示する。
- 接続開始、待機中、接続中、切断、エラーの状態遷移を UI に反映する。
- Windows ファイアウォールや mDNS 検出に関する初期設定ガイドを表示する。

### MVPで実現しないこと

- PC のクリック/キーボード入力による iPhone の完全操作。
- iPhone 内アプリを PC から直接起動すること。
- iPhone への任意入力イベント注入。
- インターネット越しのリモートミラーリング。
- 複数端末の同時ミラーリング。
- 録画、スクリーンショット、配信、ファイル転送。
- DRM 保護コンテンツやアプリ側で外部出力が制限された映像の表示保証。

## 2. できること・できないこと

| 項目 | MVPでの扱い | 補足 |
| --- | --- | --- |
| iPhone → Windows の画面表示 | 対象 | AirPlay receiver 相当の受信・復号・描画が必要。 |
| 同一 Wi-Fi 内の端末検出 | 対象 | Bonjour / mDNS によるサービス広告・検出を想定。 |
| Web UI での状態表示 | 対象 | 現在の `index.html` をデスクトップアプリ UI として発展可能。 |
| PCクリックでのiPhone完全操作 | 対象外 | iOS は通常アプリ/外部PCからの任意タップ操作を許可しないため。 |
| PCクリックでUI上の疑似操作ログ表示 | 対象 | デモUIとしては可能。ただし実機操作ではない。 |
| 録画/スクリーンショット | 将来対応 | ストリーム受信が安定した後に FFmpeg 等で追加検討。 |
| USB接続ミラーリング | 代替案 | Wi-Fi が不安定な場合の将来代替。MVP本線ではない。 |
| ブラウザだけで実機ミラーリング | 不可 | ブラウザには AirPlay 受信、mDNS広告、復号、低遅延描画のための権限/機能が不足。 |

## 3. 推奨アーキテクチャ

### 推奨構成

```text
[iPhone]
  └─ 画面ミラーリング / AirPlay 相当の送信
      ↓ Wi-Fi / LAN
[Windows Desktop App]
  ├─ Frontend: Web UI
  │   ├─ 接続状態表示
  │   ├─ 映像ビュー
  │   ├─ エラー/ガイド表示
  │   └─ 将来の録画・設定 UI
  ├─ Native Host: Tauri + Rust 推奨
  │   ├─ Windows アプリ起動/終了
  │   ├─ Firewall/権限チェック
  │   ├─ mDNS サービス広告制御
  │   ├─ Stream receiver プロセス管理
  │   └─ UI へのイベント通知
  └─ Stream Receiver Layer
      ├─ AirPlay receiver 相当プロトコル
      ├─ Bonjour / mDNS service advertisement
      ├─ 映像/音声ストリーム受信
      ├─ 復号/変換
      └─ FFmpeg またはネイティブデコーダによる描画/変換
```

### 推奨技術選定

MVP では **Tauri + Rust + AirPlay receiver 実装/ライブラリ調査 + Bonjour(mDNS) + FFmpeg** を第一候補にします。

理由:

- Tauri は Web UI を Windows デスクトップアプリに組み込みやすく、バックエンドを Rust で実装できる。
- Rust はストリーム受信、プロセス制御、低レベルネットワーク、FFmpeg 連携を型安全に実装しやすい。
- 既存の UI モックをフロントエンドとして再利用しやすい。
- Electron より軽量な配布物を目指せる。
- Node.js は開発スピードが高いが、低レベルの受信/復号/安定稼働は Rust またはネイティブプロセスに分離した方がよい。

### 技術比較

| 技術 | 役割候補 | メリット | 懸念 | MVPでの判断 |
| --- | --- | --- | --- | --- |
| Tauri | Windows デスクトップアプリのシェル | Web UI + Rust backend。軽量でネイティブ連携しやすい。 | Rust/Tauri の学習が必要。WebView 依存の差異確認が必要。 | 第一候補 |
| Electron | Windows デスクトップアプリのシェル | Chromium + Node.js を同梱し、Web 技術だけで開発しやすい。エコシステムが大きい。 | 配布サイズ・メモリ使用量が大きくなりやすい。 | 代替候補 |
| Node.js | 補助サーバー/開発用プロトタイプ | JS で HTTP/WebSocket/mDNS 周辺を試しやすい。 | 映像復号や低レベル処理はネイティブ連携が必要になりやすい。 | 補助用途 |
| Rust | ネイティブ層/受信層 | 安全性、並行処理、ネットワーク処理、Tauri 連携に強い。 | 実装難度は高め。 | 第一候補 |
| AirPlay receiver | iPhone 画面ミラーリング受信 | iPhone 標準の画面ミラーリング体験に近い。 | 仕様・暗号化・ライセンス・互換性リスクが高い。 | 調査最優先 |
| Bonjour(mDNS) | LAN内サービス広告/検出 | iPhone 側に PC 受信機を見せるために必要。 | Windows ファイアウォール/ネットワーク設定で失敗しやすい。 | 必須候補 |
| FFmpeg | ストリーム復号/変換/録画 | 映像処理の実績が大きく、将来の録画にも使える。 | ライセンス、配布、ハードウェアデコード、遅延の検証が必要。 | 受信後処理の候補 |

参考: Tauri は Web フロントエンドと Rust ロジックでクロスプラットフォームアプリを作れることを公式に説明しています。Electron は Chromium と Node.js を同梱して JS/HTML/CSS でデスクトップアプリを作るフレームワークです。Node.js はクロスプラットフォームの JavaScript runtime、FFmpeg は動画/音声のデコード・変換・ストリーム処理に使える multimedia framework です。Bonjour は Apple のサービス発見技術として公式ドキュメントがあります。

- Tauri: <https://v2.tauri.app/>
- Electron: <https://www.electronjs.org/docs/latest/>
- Node.js: <https://nodejs.org/en>
- Rust: <https://www.rust-lang.org/learn>
- FFmpeg: <https://www.ffmpeg.org/about.html>
- Bonjour: <https://developer.apple.com/bonjour/>

## 4. 各レイヤーの役割

### フロントエンド

目的: ユーザーに「接続状態」と「表示結果」を分かりやすく見せる。

担当:

- 接続待機/接続中/切断/エラーの状態表示。
- ミラーリング映像を表示するビュー領域。
- Windows ファイアウォール、同一 Wi-Fi、iPhone 側の画面ミラーリング手順を案内する。
- 受信層からのイベントを WebSocket / Tauri IPC / Electron IPC で受け取り UI を更新する。
- MVPでは「クリックで iPhone を操作できる」と誤解される UI を避ける。
- 将来機能として録画、スクリーンショット、音声設定、品質設定を追加する余地を残す。

### Windowsネイティブ層

目的: ブラウザでは扱えない OS 権限・プロセス・ネットワーク制御を担当する。

担当:

- Tauri/Electron のアプリ起動、ウィンドウ管理、トレイ常駐。
- 受信層プロセスまたは Rust モジュールの起動/停止/監視。
- Windows ファイアウォール許可のチェックとユーザー案内。
- mDNS で広告するサービス名、ポート、PIN/認証情報の管理。
- 受信層からのログ、エラー、接続メタデータをフロントエンドへ転送。
- クラッシュ時の再起動、ログ収集、診断情報の保存。

### ストリーム受信層

目的: iPhone から送られる画面ミラーリング映像を受け取り、Windows上で表示可能な形式に変換する。

担当:

- AirPlay receiver 相当のサービス広告。
- セッション開始/終了、認証、ペアリングまたは PIN 表示の検証。
- 映像/音声ストリームの受信、復号、同期。
- FFmpeg や OS ネイティブデコーダとの連携。
- 低遅延表示のためのバッファ管理。
- フレームドロップ、遅延、解像度、ビットレートのメトリクス出力。
- DRM/保護コンテンツや非対応 iOS バージョンのエラーを上位へ通知。

## 5. 次に実装するIssue一覧

### Phase 0: 調査・検証

1. **AirPlay receiver 実装方式の調査**
   - 既存OSS/商用SDK/独自実装の可否、ライセンス、iOS互換性を比較する。
2. **Bonjour(mDNS) サービス広告 PoC**
   - Windows 上で iPhone から発見できるサービス名を広告できるか検証する。
3. **Windows Firewall / LAN 設定調査**
   - 必要ポート、UDP multicast、管理者権限要否を整理する。
4. **ストリーム受信プロトコルの最小PoC**
   - iPhone が接続を開始するところまで到達できるか確認する。

### Phase 1: 表示のみMVP

5. **Tauri プロジェクトの初期化**
   - 現在の `index.html` UI を Tauri フロントエンドへ移植する。
6. **Rust ネイティブ層のイベント API 作成**
   - `receiver:start`, `receiver:stop`, `receiver:status` などを定義する。
7. **受信層プロセス管理**
   - 受信プロセスの起動、停止、ログ収集、クラッシュ検知を実装する。
8. **映像表示ビューの実装**
   - 受信した映像を WebView、native surface、または別ウィンドウに表示する方式を比較実装する。
9. **接続状態 UI の実データ連携**
   - 接続端末名、解像度、FPS、エラーを UI に反映する。
10. **切断/再接続フロー**
    - iPhone 側切断、PC 側停止、ネットワーク断に対応する。

### Phase 2: 品質改善

11. **遅延・フレーム落ち計測**
    - 実測メトリクスをログと UI に表示する。
12. **音声対応の検証**
    - 音声受信、同期、ミュート、デバイス選択を検証する。
13. **インストーラー/署名/配布設定**
    - Windows 配布、コード署名、アップデート方式を整理する。
14. **録画/スクリーンショット検証**
    - FFmpeg 連携で保存できるか検証する。

## 6. リスクと代替案

| リスク | 影響 | 対応策 | 代替案 |
| --- | --- | --- | --- |
| AirPlay receiver 仕様・暗号化・互換性が難しい | MVPが成立しない可能性 | OSS/SDK/ライセンスを最初に調査し、接続PoCを最優先にする。 | 商用SDK利用、既存AirPlay receiverアプリ連携、USB方式へ切替。 |
| Apple/iOSアップデートで動作が変わる | 継続保守コスト増 | iOSバージョン別の検証マトリクスを作る。 | 対応バージョンを限定、商用SDKに委ねる。 |
| Windows Firewall / ルーター設定でmDNSが届かない | ユーザー環境で検出不能 | 診断画面、手順ガイド、ポート確認、ログ出力を実装。 | 手動IP接続、USB接続、同梱ネットワーク診断ツール。 |
| WebViewで低遅延映像表示が難しい | 遅延・カクつき | native surface / 外部プレイヤー / GPUデコードを比較する。 | Electron + Chromium video pipeline、別ネイティブウィンドウ表示。 |
| FFmpeg 配布ライセンス/特許/コーデック問題 | 配布制約 | 使用ビルド、ライセンス、必要コーデックを事前確認する。 | Windows Media Foundation 等のOSネイティブデコーダ利用。 |
| PCクリック完全操作の誤期待 | 仕様不一致 | MVP文言で対象外と明記し、UIから完全操作表現を外す。 | 将来はアクセシビリティ/MDM/専用iOSアプリ連携など限定操作のみ検討。 |
| 商用利用時のAirPlayライセンス懸念 | リリース不可リスク | MFi/Apple関連要件、SDK契約、商標表記を法務確認する。 | AirPlay非依存の専用iOS送信アプリ方式、USBキャプチャ方式。 |

## 今後の判断基準

- **最初のGo/No-Go**: Windows から mDNS 広告し、iPhone の画面ミラーリング候補に表示できるか。
- **次のGo/No-Go**: iPhone から接続開始し、映像ストリームの最初のフレームを受信/表示できるか。
- **MVP完了条件**: 同一Wi-Fiで、表示のみのミラーリングを安定して開始・停止でき、接続エラーをユーザーに説明できること。
